# Executive Summary

The ITS Summer Internship Program currently maintains its materials across a scattered  
collection of documents, guides, and informal resources (slack, email),  with no single source of truth. This  
project will consolidate those materials into a GitHub-based, community-driven knowledge base  
— the InTurn Playbook — modeled after collaborative open handbooks pioneered by  
organizations like Dev Bootcamp and Thoughtbot.

The Playbook will be structured around the internship’s natural arc (pre-arrival through wrap-  
up), with content released in phases — much like a weekly syllabus — so that information  
reaches participants when it is timely and relevant. Critically, interns will be active contributors:  
submitting pull requests, filing issues, and participating in discussions that improve the very  
resource they are actively leveraging to support their internship.. This transforms the handbook from a static document that gets handed down into a living institution that each cohort helps build.

The primary deliverable is a fully migrated, accessible intern handbook with a defined  
governance model. The project will also establish the framework and patterns needed to extend  
the same model to a supervisor handbook, a mentor and coaches handbook, and future  
audience-specific resources in subsequent cohorts.

Most handbook projects produce a document. This project produces a practice — a way of working  
with shared knowledge that outlasts any single cohort. The interns who contribute to the InTurn  
Playbook this summer will see their work carried forward, built on, and credited in future iterations.  
That is the promise of the open handbook model, and it is worth building carefully.

# Project Details

## **Problem Statement**

The ITS Summer Internship Program’s knowledge is currently distributed across multiple  
formats and locations with no canonical home. This creates several compounding problems:

* Staff answer the same questions repeatedly across cohorts, with no mechanism to capture and share those answers at scale  
* Supervisors and mentors working with interns for the first time have no structured, reliable resource to consult  
* Institutional knowledge accumulated by each cohort evaporates at summer’s end rather than being incorporated into a shared record  
* Interns receive inconsistent onboarding experiences depending on who their supervisor is and which documents happened to be current  
* There is no process for participants to flag outdated content, propose improvements, or contribute their own experience back to the program  
* Interns gain hands-on experience with Git and GitHub — submitting pull requests, responding to review feedback, and contributing to a shared repository are core professional skills. The Playbook gives them a real context to develop those habits, not a contrived exercise.  
* Markdown keeps the content open and durable — plain text is easy to edit, renders across formats, works with any tool, and is compatible with modern AI-assisted workflows. It ensures the Playbook can grow without being locked to any single platform.

Beneath these operational problems is a missed opportunity: a well-run internship program  
should build a sense of shared community and shared ownership among its participants. A  
scattered, top-down document structure works against that. The InTurn Playbook addresses the  
operational problems and the cultural one simultaneously.

## **Project Benefits and Goals**

How will we know that the project benefits were achieved?

* A single GitHub repository serves as the canonical home for the intern handbook, with all existing content migrated, organized, and rendered as accessible HTML  
* A governance model is documented and in place: who can commit directly, how pull requests are submitted and reviewed, how issues are filed, and what the approval chain looks like  
* Content is released in phases aligned to the internship calendar (pre-arrival → orientation → weekly → wrap-up), with a hybrid time-based and milestone-gated release model  
* The rendered handbook meets WCAG 2.1 AA accessibility standards, verified through both automated tools and manual review  
* At least one intern pull request improving handbook content has been submitted and merged demonstrating the contribution model works in practice  
* A documented framework exists for extending the same model to the supervisor handbook and mentor/coaches handbook in future cohorts

### **The InTurn Playbook Model**

Inspired by Dev Bootcamp’s phase-based GitHub handbook and Thoughtbot’s open playbooks,  
 the InTurn Playbook uses GitHub as more than a storage location — it uses it as a participation platform:

* Phased content release: New material is made available on a mostly weekly schedule, aligned to where interns are in the program. Certain content is milestone-gated (e.g., wrap-up materials are not released until near the end of summer). This keeps the experience timely rather than overwhelming.  
* Pull requests as contribution: Interns who notice outdated content, want to add a resource, or have a better way to explain something can submit a PR. This is not a suggestion box — it is a real workflow with real consequence.  
* Issues as discussion: Questions, concerns, and ideas that don’t rise to a PR can be filed as GitHub Issues, creating a searchable record of community discussion that informs future iterations.  
* Separate repos per audience: The intern handbook, supervisor handbook, and mentor/coaches handbook each get their own repository. This allows access to be scoped appropriately while sharing a common structure and contributing conventions.  
* Extensible framework: The patterns established this summer — repo structure, governance model, release workflow, accessibility standards — become the template for future handbooks.

### **Audience and Access Policy**

Because the program includes participants from outside the University, and because some content is relevant during pre-arrival onboarding, the default posture for the intern handbook will be as open as possible. The project will include defining a tiered access policy as a deliverable — determining which content can be fully public, which should be U-M-internal, and what, if anything, requires restricted access. This policy will be documented as part of the repository’s governance model.

**What known challenges and concerns does this effort face?**

* Content ownership ambiguity — It may not be immediately clear who currently “owns” certain sections of the existing handbook. The project will need to surface and resolve these ownership questions early.  
* Scope discipline — The impulse to redesign and improve content while migrating it is real and must be actively managed. The guiding principle will be to migrate first, improve second.  
* Git onboarding curve — Interns with no prior Git experience need a real foundation before they can contribute effectively. Weeks one and two should include dedicated time for this.  
* Accessibility complexity — Tables, embedded media, and some legacy document formats may require significant rework to meet accessibility standards. This should be scoped and triaged early.  
* Access policy decisions — Determining what is public vs. internal vs. restricted requires buy-in from ITS leadership beyond the project team. This decision should be escalated and resolved early so it does not block content migration.  
* Sustaining the model — The Playbook only works if staff commit to the PR-and-issues workflow after the summer. Establishing that norm is as much a change management challenge as a technical one.
